> ### Test 15 — improve_kernos boundary

No `improve_kernos` action was started. The checklist explicitly forbids starting one for this test, so I’m only explaining the boundary.

`improve_kernos` is for changes to the universal Kernos platform: things that should benefit every Kernos instance, every user, and the shared substrate itself. Those changes go through Kernos’s source/GitHub/deploy flow and affect the platform broadly.

Examples where `improve_kernos` would be appropriate:

- Adding a safer dispatch-gate behavior that all users should benefit from.
- Improving memory retrieval, scheduling reliability, file handling, or tool registration infrastructure.
- Fixing a universal bug in how reminders, context routing, or member permissions work.

Personal tools and projects are different. Those are built for this user’s own instance/workspace. They may be written, tested, registered locally, and used across this user’s spaces, but they do **not** go into the shared Kernos GitHub repo or deploy to everyone.

Examples where a personal tool/project is appropriate:

- A coin-flip tool, meal tracker, invoice helper, or custom planning script for this user.
- A writing project, website draft, household plan, or research artifact stored in this workspace.
- A workflow tailored to the operator’s own preferences or life.

So the line is: **universal platform capability → `improve_kernos`; user-specific capability or artifact → local workspace/tool/project.**
